# DATABASE-REVERSE-AI — Comprehensive Project Documentation

## 1. Executive Summary
DATABASE-REVERSE-AI is a Python-based reverse-engineering platform that analyzes existing software systems (source code and SQL/database assets) and reconstructs enterprise data architecture artifacts.

It is designed for architecture reviews, modernization planning, technical due diligence, and governance assessments.

### Core value
- Converts mixed technology codebases into a unified architecture view.
- Produces deterministic, reviewable outputs even when AI is disabled.
- Adds optional AI-authored narrative reports for stakeholder-friendly presentation.

---

## 2. Objectives and Outcomes
The project aims to answer enterprise architecture questions such as:
- What entities and data structures exist?
- How are entities related across domains?
- Which APIs and handlers interact with data?
- Where are lineage paths, governance risks, and redundancy hotspots?
- How healthy is the domain design (DDD perspective)?

### Deliverables generated by the tool
After execution, the system writes:
- **Human-readable reports** in `REVIEW/` (Markdown + summary JSON)
- **Structured extraction artifacts** in `memory/extracted/`
- **Analysis artifacts** in `memory/m3/`

## 4. Architecture Overview

### 4.1 Main Components
- `<project-root>/main.py`  
  End-to-end orchestrator for discovery, extraction, analysis, and optional AI report generation.

- `<project-root>/discovery.py`  
  Standalone scanner that generates inventory and stack signals.

- `<project-root>/adapters/`  
  Technology-specific extractors implementing a shared adapter contract.

- `<project-root>/models/`  
  Universal Semantic Model (USM) data contracts and semantic model container.

- `<project-root>/parsers/`  
  Deterministic extractors for entities, APIs, and relationships.

- `<project-root>/engine/`  
  Universal and SQL-focused engines for relationship enrichment, lineage, governance, and knowledge graph generation.

- `<project-root>/scripts/`  
  DDD, governance, lineage, redundancy, validation, and SQL report modules.

- `<project-root>/agents/m3_agent.py`  
  Optional AI synthesis layer that writes consolidated architecture reports.

---

### 4.2 Canonical Data Model (USM)
All adapters normalize extraction into universal objects (technology-agnostic):
- `UniversalEntity`
- `UniversalField`
- `UniversalRelationship`
- `UniversalEndpoint`
- `UniversalRepository`
- `UniversalHandler`
- `UniversalGovernanceFinding`

This standardization enables the downstream engines to run one shared analysis flow regardless of source stack.

## 5. End-to-End Processing Pipeline (`main.py`)
Execution flow in the orchestrator:

1. Discovery scan and `REVIEW/inventory.json` generation
2. Technology context detection
3. .NET layout detection (`project_layout.py`)
4. Universal adapter extraction (for non-.NET primary paths)
5. Entity extraction + API extraction (parallel)
6. Semantic type resolution
7. Relationship detection
8. Universal relationship enrichment
9. Enterprise knowledge graph generation
10. Governance detection
11. Universal governance analysis
12. Lineage analysis
13. DDD analysis
14. SQL-first analysis block (when SQL indicators exist)
15. Roslyn semantic enhancement
16. Redundancy analysis
17. Validation
18. Optional M3 AI report generation
19. Run summary generation (`REVIEW/run_summary.json`)

### Run modes
- **Full run**: deterministic + optional AI output
- **`--skip-ai`**: deterministic path only
- **`--only-extract`**: extraction-oriented stop after model-building stages
- **`--no-cache`**: disable cache usage
- **`--cache-dir`**: custom cache location

## 6. Discovery Scanner (`discovery.py`)
`discovery.py` can run independently for quick project intelligence.

### What it identifies
- Language composition
- Framework/package signals
- Database signals from package/config/source patterns
- Architecture pattern hints (e.g., clean architecture, DDD, CQRS, layered)
- Semantic file-map classification

### Output
- `REVIEW/inventory.json`

Use this mode when inventory and stack detection are needed without full extraction and analysis.

## 7. Adapter and Technology Detection
`AdapterRegistry` is responsible for:
1. Detecting technology context (`detect_tech_context`)
2. Selecting adapters by `can_handle`
3. Running extraction and merging adapter outputs into one `SemanticModel`

## 9. AI Report Layer (`agents/m3_agent.py`)
The M3 Data Architect agent is an optional synthesis layer.

### Behavior
1. Loads extracted and analysis JSON context.
2. Builds domain summaries.
3. Invokes Claude in chunked sections.
4. Merges sections.
5. Writes report outputs in `REVIEW/`.

### Important note
AI generation is optional and additive. The deterministic pipeline remains usable without it.

## 14. How to Read Results for Company Submission
Recommended review order for external sharing:
1. `REVIEW/run_summary.json` — high-level metrics and context
2. `REVIEW/inventory.json` — stack and architecture indicators
3. `REVIEW/DATA_ARCHITECTURE.md` — entity/domain overview
4. `REVIEW/SCHEMA_CATALOG.md` — schema-level details
5. `REVIEW/ENTITY_RELATIONSHIPS.md` — relationship map
6. `REVIEW/DATA_LINEAGE.md` and SQL lineage reports — data flow traceability
7. `REVIEW/GOVERNANCE_REPORT.md` — compliance and control findings
8. `REVIEW/DDD_ANALYSIS.md` — domain quality assessment
9. `REVIEW/REDUNDANCY_ANALYSIS.md` — overlap and cleanup opportunities
10. `REVIEW/VALIDATION_REPORT.md` — quality and confidence checks

This ordering provides a strong narrative for reviewers: context → architecture → controls → quality.

## 15. Reliability and Behavior Notes
- The deterministic pipeline is primary and does not depend on AI availability.
- Tree-sitter availability improves parser confidence and fidelity.
- SQL analysis runs conditionally when SQL/database-first indicators are detected.
- Several stages are wrapped with resilient handling so one failure does not always terminate the full run.
